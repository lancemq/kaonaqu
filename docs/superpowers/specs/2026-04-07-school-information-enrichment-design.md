# 学校信息大范围补全设计

日期：2026-04-07
主题：学校库从“可浏览”升级为“基础可用 + 重点可判断”的信息资产

## 目标

将当前学校库从“已有学校名录和部分档案”升级为一套可持续补全的学校信息系统。

本次设计的核心目标不是一次性把所有学校都补到最深，而是建立一套可以长期扩展的补全策略，让学校库同时具备：

- 全库基础可用
- 重点学校信息可判断
- 标签体系可筛选、可比较
- 数据补全流程可持续更新

本次优化聚焦两个优先方向：

- 基础详情补全
- 标签与结构化补全

不把“质量控制和持续更新”作为第一优先级输出目标，但必须作为方案的一部分一起设计。

## 用户目标

当前学校库需要同时服务两类用户：

1. 快速查学校的人
2. 需要横向比较学校的人

要支持这两类使用场景，学校信息不能只停留在“名称 + 区域 + 少量介绍”，而需要逐步形成两层能力：

- 基础档案：让每一所学校至少像一份完整的基础资料卡
- 深度判断：让高关注学校可以被拿来比较、筛选和判断

## 设计方向

本次采用“双层补全”策略，而不是单一策略。

### 1. 全库基础补全

目标：

- 让所有学校先达到“基础可用”
- 解决学校库里“有名字但内容很空”的问题

特点：

- 字段尽量客观
- 来源尽量稳定
- 支持批量化规则补全

### 2. 高关注学校深度补全

目标：

- 让用户最常看的学校优先达到“可判断、可比较”
- 让学校详情页和筛选标签真正有决策价值

特点：

- 允许更多人工校准
- 允许更高成本的结构化归纳
- 重点补充判断型字段，而不只是客观字段

## 范围策略

第一阶段采用“分层覆盖”而不是统一深度：

- 全库学校：先补基础字段
- 高关注学校：在基础字段之上继续补深度字段和标签

高关注学校的首批范围按“流量 / 用户关注度”选择，而不是平均覆盖全市所有代表学校。

这样做的理由：

- 更符合真实用户需求
- 能更快提升学校频道的感知价值
- 不会因为追求平均覆盖而把重点学校做得很浅

## 字段分层

### 1. 全库基础字段

适合大范围补全的字段：

- 学校名称
- 区域
- 学段
- 办学性质 / 学校类型
- 地址
- 官网
- 联系方式
- 学校简介短句
- 基础招生说明
- 基础标签
- 数据来源
- 更新时间

这部分字段的目标不是“写得多”，而是保证每所学校至少具备一份完整、可读、可信的基础档案。

### 2. 高关注学校深度字段

仅对高关注学校优先补全的字段：

- 学校定位句
- 培养方向
- 高价值标签组合
- 适合人群
- 申请建议 / 关注点
- 更完整的招生口径
- 学校亮点摘要
- 需要解释和归类的特色信息

这部分字段的目标不是“看起来丰富”，而是“对比较和判断有帮助”。

### 3. 第一阶段不建议大规模补全的字段

以下内容不适合作为第一阶段的全库目标：

- 升学成绩类结论
- 主观口碑判断
- 没有稳定来源支撑的“适合某类孩子”
- 变化较快且难维护的细颗粒度收费、班型和课程细节

这些内容可以作为后续专题化补充，但不适合在第一阶段硬铺到全库。

## 标签体系

标签必须分层，否则会很快失控。

### 1. 稳定标签

稳定标签指长期变化较小、适合批量规则化补全的标签：

- 公办
- 民办
- 双语
- 外籍学校
- 初中
- 高中
- 完全中学
- 寄宿
- 九年一贯
- 外语特色
- 国际课程
- 示范性高中
- 市重点
- 区重点

规则：

- 稳定标签允许批量补全
- 稳定标签优先依据学校名、办学类型、公开口径和已有结构字段生成

### 2. 判断标签

判断标签指带有归纳和解释性质、适合比较但不适合无证据批量推断的标签：

- 科创竞赛
- 人文综合
- 艺术特色
- 寄宿管理
- 贯通培养
- 教研课程强
- 德育活动强

规则：

- 判断标签不作为第一阶段全库必填字段
- 判断标签优先在高关注学校中补全
- 判断标签需要更严格的来源和校准规则

## 补全流水线

本次不建议写成一个“大而全脚本”，而建议拆成 4 段流水线。

### 1. 学校分层

先将学校分为两个池：

- 全库学校池
- 高关注学校池

高关注学校池由一份可维护名单驱动，可根据流量、用户关注和业务判断持续扩展。

### 2. 基础信息采集

对全库学校，优先采集稳定来源：

- 学校官网
- 教育局 / 教委公开信息
- 招生简章
- 学校概况页
- 第三方资料仅做补充，不作为唯一依据

输出目标：

- 基础详情文本
- 来源信息
- 可结构化字段候选

### 3. 规则化结构提取

把采集到的文本转成结构字段候选：

- 学段
- 办学性质
- 地址 / 电话 / 官网
- 基础简介
- 基础标签候选

这一步的原则是“先生成候选，不直接假定完全准确”。

### 4. 高关注学校人工校准

只对高关注学校追加人工提纯步骤：

- 调整定位句
- 确认深度标签
- 归纳培养方向
- 补充适合人群
- 补充申请建议
- 校准亮点摘要

最终形成：

- 全库学校靠规则打底
- 高关注学校靠规则加人工提纯

## 页面消费策略

数据补全不仅要能存，还要能被页面稳定消费。

### 1. 学校卡片

学校卡片优先消费这些信息：

- 学校名
- 区域 / 学段 / 办学性质
- 一句定位句
- 2 到 4 个高价值标签
- 简短更新时间 / 数据口径

卡片目标是：

- 让用户快速横向比较
- 支持筛选后快速缩范围

### 2. 学校详情页

学校详情页优先消费这些区块：

- 基础识别
- 学校定位
- 招生与入学线索
- 培养方向 / 特色标签
- 适合人群 / 申请建议
- 数据来源与更新时间

详情页目标是：

- 让用户形成判断，而不是只看一段冗长介绍

### 3. 页面消费原则

- 基础字段不够时宁可留空，也不写泛话
- 深度字段优先服务高关注学校
- 标签和文案不要彼此重复
- 所有判断型字段尽量能追溯到来源或归类规则

## 质量控制

虽然本次不是先做质检系统，但补全方案必须从第一天就带上质检。

至少需要以下 4 类检查：

### 1. 字段完整性检查

检查哪些学校缺基础字段，哪些高关注学校缺深度字段。

### 2. 标签冲突检查

识别标签之间的明显冲突或结构字段与标签不一致的情况。

### 3. 来源覆盖检查

对判断型字段，若没有来源或规则支撑，不进入正式展示数据。

### 4. 人工抽检清单

每轮补全后优先抽检：

- 高关注学校
- 新增学校
- 新增标签规则影响到的学校集合

## 数据结构建议

本次不要求立刻确定最终 schema 细节，但建议在现有学校结构上明确两层字段用途：

### 1. 基础字段组

继续保留并补全：

- `name`
- `districtId`
- `districtName`
- `schoolStage`
- `schoolStageLabel`
- `schoolType`
- `schoolTypeLabel`
- `address`
- `phone`
- `website`
- `schoolDescription`
- `admissionNotes`
- `source`
- `updatedAt`

### 2. 深度字段组

优先用于高关注学校：

- `schoolHighlights`
- `suitableStudents`
- `applicationTips`
- `features`
- `tags`
- `trainingDirections`

建议未来为判断型内容补充清晰来源或归类依据，但第一阶段不强求所有字段一次到位。

## 实施边界

本次设计不包含以下内容：

- 所有学校一次性深度写全
- 全量 OCR / 文档解析系统
- 升学成绩和口碑类主观判断系统
- 复杂推荐算法
- 所有标签都自动化推断

本次重点是先把学校库做成：

- 能批量打底
- 能重点做深
- 能持续扩展

## 结论

学校信息大范围补全应采用“双层方案”：

- 全库先补基础详情，解决“每所学校至少可用”
- 高关注学校再做深度补全，解决“重点学校真的能判断”
- 标签分为稳定标签和判断标签两层
- 流水线分为分层、采集、提取、人工校准四段
- 页面消费按卡片与详情页分层
- 质检机制和补全流程同时设计

这不是一次性补数据，而是在为学校库建立一条长期可维护的成长路径。
